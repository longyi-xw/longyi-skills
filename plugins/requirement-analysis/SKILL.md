# Requirement Analysis（需求分析 · 产品需求转研发视角）

以一名**优秀产品经理**的身份，读懂一份（或一组）需求文档，然后把它翻译成研发工程师真正能用来干活的东西。

**核心定位**：本技能的价值**不在于产出一份更专业的需求文档**，而在于做「翻译 + 结构化 + 可视化」——把产品同学的语言、业务背景、零散功能点，转成**流程化、架构化、程序化**的表达，让拿到它的工程师（不限岗位）能快速回答三个问题：「这个项目到底要做什么」「大致该怎么搭」「哪里是硬骨头」。

最终交付一份结构化 Markdown 文档，写到工作目录，文件名 `REQUIREMENT_ANALYSIS.md`（若已存在则改名为 `REQUIREMENT_ANALYSIS_<时间戳>.md`，绝不覆盖用户已有文件），并用 `present_files` 呈现。

---

## 五大交付物（本技能必须产出的内容）

无论输入是一份正式 PRD 还是几段零散需求描述，最终都围绕这五件产物组织。用户可只要其中几件，但默认全给。

1. **📌 项目简介**（精准、一段话级别）——这个项目是什么、给谁用、解决什么问题、业务/商业价值。让人 30 秒建立全局认知。
2. **📋 核心需求说明**（完整、易理解）——把功能需求 + 非功能需求，用去黑话、结构化、工程师友好的方式讲清。不是照抄 PRD，是**重新组织与翻译**。
3. **🔀 项目流程图**（非常清晰）——核心业务流程 / 用户旅程 / 关键交互链路，用 Mermaid 画出来。
4. **🏗️ 架构设计图**——从需求**推导**出的系统架构建议（分层 / 模块 / 服务 / 数据 / 外部依赖），Mermaid 呈现。明确标注这是「基于需求的推断，以团队技术评审为准」。
5. **⚠️ 实施难点总结**（研发视角）——技术难点、风险点、需求里的模糊/缺口、性能与数据规模挑战、跨岗位协作点。这是产品文档里通常**没有、但工程师最想要**的部分。

---

## 核心原则（贯穿全程，优先级最高）

1. **忠于文档，不编造需求。** 文档里没有、也没被用户确认的功能或规则，不要臆造。需要补的地方，以「待与产品/业务确认」的形式提出，而不是自己脑补一个塞进去。

2. **「产品语言 → 研发语言」是本技能的核心价值。** 产品文档里的「赋能」「打通」「一键」「智能推荐」背后，对应的是具体的接口、状态、数据流、算法。你的工作就是把这层翻译做到位。翻译对照与常见黑话见 `references/extraction-and-translation.md`。

3. **架构与难点是「推断/建议」，不是定论。** 你不是在替团队做技术选型拍板，而是给出一个合理的起点和思考框架。所有架构图、选型建议、难点判断都要标注「供参考，以技术评审为准」，把决定权留给研发团队。

4. **诚实暴露需求的歧义与缺口。** 需求文档最大的坑是「看着完整，一做就发现没定义」。主动找出：边界条件没说清的、异常流程缺失的、非功能需求（性能/并发/安全/兼容）只字未提的、多个需求互相矛盾的。这些「坑」的价值往往超过复述需求本身。检查清单见 `references/extraction-and-translation.md`。

5. **面向多角色，标注每部分归谁。** 服务对象不限于前端。先从文档内容推断涉及哪些研发岗位（前端 / 后端 / 移动端 / 算法 / 数据 / 测试 / 运维 / 嵌入式…），再在核心需求、架构、难点中标注「这块主要与 XX 相关」，让不同岗位各取所需。

6. **程序员友好的表达。** 多用表格和 Mermaid 图，少用大段散文。术语、标识符保留英文原文。默认输出中文（跟随用户语言）。

## 工作流程

按五个阶段推进。**先收集与定位，再提炼翻译，最后可视化产出。**

### 阶段 0 · 收集输入与明确范围

开始前确认三件事，缺什么一次性列清让用户补齐：

1. **有哪些输入**：正式文档（立项书 / MRD / PRD / 原型标注 / 接口约定 / 评审纪要），还是零散的需求描述？多份文档要说明各自是什么。
   - 若用户只丢来几段需求文字、没有正式文档，**照样能做**——把这几段当作需求来源，只是在文档里注明「基于用户口述需求，非正式文档」，并让「待确认」区更饱满。
2. **面向哪些研发角色**：**先自动推断**（见阶段 2），只有当文档明显跨多个岗位、且无法判断主次时才问用户。
3. **要哪几件产物**：默认五件全给；用户若说「只要流程图和难点」就按需裁剪。

> 若输入文档是需要读取的文件（docx / pdf / 图片版 PRD / xlsx 等），且内容不在上下文里，先按 `file-reading` 技能的指引选对读取方式，别对二进制文件直接 `cat`。图片/扫描版 PRD 用视觉方式读。

### 阶段 1 · 读取与信息提炼

通读输入，从中提炼出以下要素（对照 `references/extraction-and-translation.md` 的「按文档类型提炼要点」）：

- **业务目标 / 商业价值**：为什么要做这个项目？（立项文档、MRD 的核心）
- **目标用户 / 使用场景**：谁在什么场景下用？
- **核心功能清单**：要做哪些功能？（PRD 的主体）
- **非功能需求**：性能、并发量、数据规模、安全、兼容性、可用性——**这些常常散落各处甚至缺失，要专门找**。
- **约束与依赖**：技术约束、外部系统依赖、合规要求、时间/里程碑。
- **明确的边界**：文档里说了「不做什么」的部分（很有价值，别忽略）。

提炼时**边读边记疑点**：哪里含糊、哪里矛盾、哪里缺定义，攒到阶段 3 汇总。

### 阶段 2 · 需求解构与翻译（产品语言 → 研发语言）

这是本技能的核心动作。做三件事：

**① 识别涉及的研发角色。** 根据功能特征判断需要哪些岗位参与，例如：
- 有界面/交互/H5/小程序 → 前端；原生 App → 移动端
- 有数据存储/接口/业务逻辑 → 后端；有实时通信 → 后端 + 前端（WebSocket/WebRTC）
- 有「推荐 / 识别 / 预测 / 大模型」→ 算法；有「报表 / 大屏 / 数仓 / 埋点」→ 数据
- 有「私有化部署 / 高可用 / 容器化」→ 运维；有硬件/设备交互 → 嵌入式

**② 把产品表述翻译成工程概念。** 对照 `references/extraction-and-translation.md` 的「产品黑话 → 工程语义」对照表。例如「一键导出」= 后端生成任务 + 异步处理 + 前端轮询/推送 + 文件下载。翻译时点明背后隐含的**状态、接口、数据流、边界**。

**③ 归类为功能需求 vs 非功能需求。**
- 功能需求：系统要「做什么」，可拆成「用户故事 / 用例」的形式。
- 非功能需求：系统要「做得多好」——性能、容量、并发、安全、可维护性。文档没写的，在这一步就要立成「待确认」条目（例如「PRD 未说明并发量级，直接影响架构选型，需确认」）。

### 阶段 3 · 推导流程与架构

基于解构结果，**推导**（不是照抄，文档一般也没有）出流程和架构：

**梳理核心流程**：把最主要的一两条业务主线，画成流程图（用户旅程 / 状态流转 / 系统时序）。复杂项目可拆多张，别把所有分支塞进一张图。

**推导系统架构**：从需求反推一个合理的架构起点——分几层、有哪些模块/服务、数据怎么存、依赖哪些外部系统。**这是建议不是钦定**：给出主流、稳妥的选型，并说明「为什么这样分」，同时留出「团队可能调整」的空间。若用户/文档已指定技术栈，就顺着它推导，不要另起炉灶。

> ⚠️ 每张架构图、每个选型建议下方都标注：`> 供参考，具体选型以团队技术评审为准`。

### 阶段 4 · 汇总实施难点（研发视角）

站在工程师立场，把「做这个项目会在哪里卡住」讲清楚。至少覆盖：

- **技术难点**：文档里轻描淡写、实际很硬的点（如「实时同步百万级数据」「跨端一致性」）。
- **需求缺口/歧义**：阶段 1–2 攒下的所有「待确认」，在这里汇总成清单，标明**每一项会阻塞谁、影响什么决策**。
- **性能与规模挑战**：数据量、并发、时延要求带来的挑战（可结合使用者在时序数据库、大规模轨迹数据等领域的经验，如相关）。
- **跨岗位协作点**：哪些功能需要多个岗位联调、接口约定要提前拉通。
- **风险与依赖**：外部系统、第三方、合规、时间风险。

每条难点尽量给一句「建议方向」，而不只是抛出问题。

### 阶段 5 · 产出文档

读取 `references/output-template.md`，按模板生成 `REQUIREMENT_ANALYSIS.md`，写到工作目录。

**文件已存在处理**：先检查同名文件，若存在则新文件命名为 `REQUIREMENT_ANALYSIS_YYYYMMDD_HHMMSS.md`（用 `date +%Y%m%d_%H%M%S` 取时间戳），并告知用户「检测到已有文档，已另存为 …，未覆盖原文件」。**任何情况下不覆盖用户已有文件。**

**Mermaid 图**：流程图用 `flowchart` / `stateDiagram-v2` / `sequenceDiagram`，架构图用 `graph TD` / `graph LR`。按项目实际画，不要照搬模板样例数据。

生成后用 `present_files` 呈现文档，并在对话里给一段 3–5 句口头摘要（这个项目是什么、最关键的一两个需要研发注意的点）。

## 关键提醒

- **翻译到位、可视化清晰，是本技能与「复述一遍 PRD」的分水岭。** 如果产出读起来还是产品腔、还是一堆功能点罗列，就说明翻译没做够。
- **「待确认」清单是硬通货。** 一个好的需求分析，往往靠它暴露的坑赢得研发信任。宁可多标几个待确认，也别假装需求很完整。
- **不越位做技术拍板。** 架构和选型给「合理起点 + 理由」，把评审权留给团队。
- **按输入丰俭由人。** 有完整 PRD 就做深；只有几句话就做浅，并把不确定性显式标注出来，绝不用编造填满模板。
- **语言与术语**：正文中文，代码/接口/技术名词保留英文。
