## 一、按文档类型的提炼要点

不同文档回答不同问题，提炼时各有侧重。一个项目可能只给其中一种，也可能几种都有。

| 文档类型 | 核心回答 | 重点提炼什么 | 常见坑 |
|---------|---------|------------|--------|
| **项目立项文档 / 商业计划** | 为什么做、投入产出 | 业务目标、目标市场、商业价值、里程碑、预算/人力约束 | 只有愿景没有功能细节，别把「愿景」当「需求」 |
| **市场需求文档（MRD）** | 用户是谁、痛点是什么 | 目标用户画像、使用场景、痛点、竞品参照、优先级 | 偏市场语言，功能常常还没拆到可实现的粒度 |
| **产品需求文档（PRD）** | 具体要做什么功能 | 功能清单、交互流程、字段/规则、边界条件、状态定义 | 「主流程写得细、异常流程一笔带过」是通病 |
| **原型 / 交互稿标注** | 界面长什么样、怎么交互 | 页面结构、控件、跳转关系、状态切换、空/错/加载态 | 静态图看不出动态逻辑，动效与实时性要额外确认 |
| **接口 / 数据约定** | 系统间怎么对接 | 接口定义、字段、协议、鉴权、数据格式 | 常与 PRD 不同步，以哪个为准要问清 |
| **需求评审纪要 / 变更记录** | 最新结论是什么 | 已拍板的决定、被否掉的方案、遗留问题 | 纪要往往最新，和旧 PRD 冲突时通常以纪要为准 |

**提炼时始终追问这六项**（很多文档会漏其中几项，漏的就是「待确认」）：
- 业务目标：这个项目/功能存在的理由。
- 目标用户与场景：谁、在什么情况下、为什么用。
- 功能范围：做什么 + **明确不做什么**（scope 边界极其重要）。
- 非功能需求：性能、并发、数据量、安全、兼容、可用性——**专门找，大概率散落或缺失**。
- 约束与依赖：技术栈限制、外部系统、合规、时间。
- 验收标准：怎么算「做完了」「做对了」。

## 二、产品黑话 → 工程语义 翻译对照

产品文档里的一个词，背后往往是一整条工程链路。翻译的目标：**点破隐含的接口、状态、数据流、边界、异常**。下表是常见模式，实际翻译要结合具体上下文，不要生搬。

| 产品表述 | 背后的工程语义（翻译时点破这些） |
|---------|-------------------------------|
| 「一键 XX」（导出/生成/同步） | 触发动作 + 后端任务（可能异步）+ 进度反馈（轮询或推送）+ 结果交付 + 失败重试。「一键」在前端，复杂度在后端。 |
| 「实时」 | 到底是「秒级刷新」还是「真·实时」？前者轮询即可，后者要 WebSocket/SSE/长连接。时延要求直接决定技术方案，必须量化。 |
| 「智能推荐 / 智能匹配」 | 是规则引擎、检索排序，还是模型？训练数据从哪来？冷启动怎么办？先问清「智能」的具体含义，别默认要上算法。 |
| 「支持海量 / 大规模数据」 | 量级是多少？万、百万、还是亿？分页、索引、分库分表、时序库、缓存的取舍全看这个数。**必须量化。** |
| 「打通 / 对接 XX 系统」 | 有没有现成接口？协议是什么？鉴权方式？对方稳定性如何？失败降级怎么处理？跨系统就有一致性问题。 |
| 「同步 / 数据一致」 | 强一致还是最终一致？冲突怎么解？双向还是单向？这决定是简单覆盖还是要做冲突合并。 |
| 「灵活配置 / 自定义」 | 配置项的边界在哪？存哪、怎么下发、要不要热更新？「灵活」越大，抽象成本越高，容易过度设计。 |
| 「秒开 / 流畅 / 不卡」 | 量化成指标：首屏时间、FPS、接口 P99 时延。不量化就无法验收，也无法判断难度。 |
| 「权限管理」 | RBAC 还是 ABAC？粒度到菜单、按钮还是数据行？多租户吗？权限模型选错后期极难改。 |
| 「审批流 / 工作流」 | 固定流程还是可配置流程引擎？会签/或签/驳回/加签都要吗？状态机要提前设计清楚。 |
| 「消息通知 / 触达」 | 站内信、Push、短信、邮件哪几种？实时还是定时？到达率要求？是否需要消息队列削峰。 |
| 「大屏 / 可视化 / 看板」 | 数据实时性？数据量？渲染方案（Canvas/WebGL/图表库）？后端要不要预聚合。 |
| 「离线可用」 | 本地存储方案、同步策略、冲突解决——离线是移动端/桌面端的复杂度放大器。 |

**翻译动作要点**：读到一个功能，问自己「要实现它，前端做什么、后端做什么、数据存哪、异常怎么办、边界在哪」，把答案写进核心需求说明，这就是翻译。

## 三、需求歧义与缺口检查清单

逐项对照输入，命中的都记为「待确认」。这份清单是暴露「看着完整、一做就塌」的利器。

**功能完整性**
- [ ] 每个功能的**异常流程**是否定义？（网络失败、数据为空、并发冲突、超时）
- [ ] **边界条件**是否明确？（最大长度、数量上限、空值、极值、非法输入）
- [ ] **状态流转**是否闭环？（有没有进得去出不来、或缺失的中间态）
- [ ] 有没有**互相矛盾**的需求？（不同章节/文档对同一功能说法不一）

**非功能需求（最常缺失）**
- [ ] **性能指标**是否量化？（响应时间、吞吐、并发用户数）
- [ ] **数据规模**是否给出量级？（总量、增速、单次查询范围）
- [ ] **安全与合规**是否提及？（鉴权、数据加密、隐私、审计、等保/GDPR 等）
- [ ] **兼容性**范围？（浏览器、系统版本、设备、屏幕、网络环境）
- [ ] **可用性要求**？（SLA、容灾、降级策略）

**边界与依赖**
- [ ] **不做什么**是否说清？（scope 边界，防止无限膨胀）
- [ ] **外部依赖**是否明确？（第三方接口、上游系统、数据源的可用性与稳定性）
- [ ] **数据来源与初始化**？（历史数据迁移？种子数据？谁提供）
- [ ] **权限与角色**是否覆盖全？（不同角色能看/能做什么，是否有遗漏角色）

**验收与运营**
- [ ] **验收标准**是否可度量？（怎么算做对了）
- [ ] **上线与运维**考虑了吗？（部署方式、监控、回滚、数据备份）
- [ ] **国际化 / 多语言 / 多时区**是否需要？

**记录格式建议**：每条待确认写成「【待确认】<问题描述> —— 影响：<会阻塞谁/影响什么决策>」，让研发一眼看出优先级。例如：
> 【待确认】PRD 未说明轨迹数据的并发写入量级 —— 影响：直接决定是否需要时序数据库与分片策略，架构评审前必须确认。
